## I. Hybrid Structure

### 1. Open Core (100% Transparent)

Remain open-source:

| Component | Status |
|-----------|--------|
| Cryptographic core (Kyber, Dilithium, Hybrid) | Open |
| Handshake + session protocol | Open |
| Metadata protection layer | Open |
| ELARA transport | Open |
| Base SDK | Open |
| Formal specification | Open |

**Goals:** Auditability, academic adoption, community trust, standard candidacy.

### 2. Commercial Layer (Enterprise & Regulated Markets)

| Offering | Description |
|---------|-------------|
| **Enterprise Control Plane** | Key lifecycle dashboard, compliance reporting UI, audit analytics, centralized policy engine, role-based key governance |
| **Managed SaaS** | Hosted B4AE relay cluster, optional key escrow (compliance mode), SLA-backed deployment, monitoring & uptime guarantee |
| **Compliance Pack** | GDPR toolkit, HIPAA deployment profile, ISO 27001 control mapping, forensic export tools |

---

## II. License Strategy

| Layer | License |
|-------|---------|
| **Open Core** | MIT / Apache 2.0 (unchanged) |
| **Enterprise** | Commercial license agreement, subscription-based |

**Critical:** Do not move open core to source-available. Crypto projects lose trust when licenses change aggressively.

## VII. Trust Preservation Rules

Crypto hybrid projects fail due to:

- Monetization too early
- Core locked down
- Marketing overclaim
- No public audit

**Ensure:**

- [ ] Audit report published
- [ ] Bug bounty program
- [ ] Transparent changelog
- [ ] Clear governance
